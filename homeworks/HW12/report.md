# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет:** использован файл `S12-hw-dataset.csv` с почасовым временным рядом `target`.
- **Постановка:** прогноз следующего значения ряда (`horizon = 1`).
- **Разбиение:** выполнен строгий `temporal split` в пропорции 60% / 20% / 20%.
- **Подходы:** сравнены четыре обязательных эксперимента: `B1` (`naive-last`), `B2` (`moving-average`), `B3` (`ridge-lag-features`), `R1` (`gru-forecast`).
- **Лучший подход:** по validation лучшей оказалась `GRU`.

## 2. Среда и воспроизводимость

- **Python:** 3.11+
- **Библиотеки:** `pandas`, `numpy`, `matplotlib`, `scikit-learn`, `torch`
- **Устройство:** `cuda` при наличии, иначе `cpu`
- **Seed:** 42
- **Как запустить:** открыть `HW12.ipynb` и выполнить `Run All`

## 3. Данные и постановка задачи

- **Датасет:** `S12-hw-dataset.csv`
- **Целевая колонка:** `target`
- **Частота временного ряда:** 1 час
- **Горизонт прогноза:** 1 шаг
- **Размер датасета:** 4320 наблюдений
- **Диапазон дат:** 2025-01-01 00:00:00 — 2025-06-29 23:00:00
- **Есть ли дополнительные внешние признаки:** нет; использованы только исторические значения `target` и календарные признаки, извлечённые из `date`.
- **Комментарий:** в ряде заметна выраженная внутрисуточная структура, присутствует повторяемость паттернов и умеренная нестационарность. Пропусков в исходном датасете нет.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Как именно выполнено:** последовательное разбиение по времени без перемешивания.
- **Какие доли использовались:** 60% train, 20% validation, 20% test.
- **Почему `random split` здесь некорректен:** при случайном перемешивании модель получает информацию из будущих моментов времени, что создаёт утечку данных и завышает метрики.

### 4.2. Признаки для baseline-моделей

- **Какие лаги использовались:** `lag_1`, `lag_7`, `lag_14`
- **Какие rolling-признаки использовались:** `rolling_mean_7`, `rolling_std_7`
- **Какие календарные признаки использовались:** `hour`, `day_of_week`, `is_weekend`
- **Как обрабатывались пропуски после построения лагов/rolling:** строки с начальными `NaN` удалялись после построения признаков.
- **Использовалось ли масштабирование и где оно обучалось:** `StandardScaler`, причём `fit` выполнялся только на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (`naive-last`):** прогноз равен последнему известному значению.
- **B2 (`moving-average`):** прогноз равен среднему по последним 7 шагам.
- **B3 (`ridge-lag-features`):** `Ridge` на лаговых, rolling- и календарных признаках.
- **R1 (`gru-forecast`):** `GRU` на оконном представлении последовательности признаков.

Дополнительно:

- **Основная метрика выбора лучшей модели:** `MAE` на validation
- **Дополнительные метрики:** `RMSE`, `MAPE`
- **`window_size` для `GRU`:** 7
- **Batch size:** 64
- **Максимальное число эпох:** 30
- **Optimizer:** `Adam`
- **Learning rate:** 0.001
- **Как сохранялась лучшая модель:** сохранялся `state_dict()` модели с лучшим значением validation MAE.

## 7. Анализ

Наивный baseline в этом ряде оказался неожиданно сильным, потому что соседние значения тесно связаны между собой. Скользящее среднее с окном 7 шагов сработало хуже: оно чрезмерно сглаживает локальные колебания и запаздывает относительно быстрых изменений ряда. Ridge использует лаговые и календарные признаки более осмысленно, но ограничена линейной формой связи. GRU получает на вход последовательность из нескольких временных шагов и может учитывать зависимость между признаками внутри окна. Именно поэтому она показала лучшую validation MAE.

При этом тестовая часть получилась сложнее validation, что видно по росту тестовых метрик. Это нормальная ситуация для временных рядов: распределение в будущем может смещаться относительно прошлого. Потенциальные утечки в такой задаче возникают при неправильном построении rolling-признаков без `shift(1)`, при случайном перемешивании данных и при масштабировании на всём датасете сразу. В работе эти риски устранены: признаки строятся только из прошлого, split хронологический, scaler обучается только на train.

Выбор лучшей модели по validation принципиален. Если выбирать по test, тест перестаёт быть честной финальной проверкой и превращается в часть процесса настройки. Для временных рядов это особенно критично.

## 8. Итоговый вывод

Для такого ряда сначала обязательно нужно строить простые baseline-модели и только потом переходить к нейросетям. В данном решении лучшей по validation стала `GRU`, однако интерпретируемая модель на признаках (`Ridge`) остаётся полезным сильным базовым вариантом. Главный вывод из работы — в задачах временных рядов хронологический порядок данных нельзя нарушать, а все преобразования должны обучаться только на тренировочной части.
